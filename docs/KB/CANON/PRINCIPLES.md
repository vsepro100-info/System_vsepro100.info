# Принципы работы и взаимодействия

## Режим минимальных инструкций
- Архитектор публикует:
  - итоговые решения
  - конкретные команды для тестировщика
  - критерии проверки результата
- Архитектор не обязан объяснять ход мыслей и внутреннюю логику.
- Объяснения и теория предоставляются только по прямому запросу тестировщика.
- Тестировщик выполняет только явно выданные команды.
- Стоп-фраза тестировщика «Стоп. Это не та корова.» является обязательным сигналом немедленной остановки.

## Канонические принципы взаимодействия

### Принцип инженерной ответственности Архитектора
- Архитектор принимает архитектурные и технические решения.
- Архитектор не перекладывает выбор на тестировщика.
- Архитектор публикует итоговое решение (канон режима минимальных инструкций), а не набор вариантов.

### Принцип «Один шаг за раз»
- Архитектор выдаёт только одну активную команду.
- Тестировщик выполняет только текущий шаг.
- Следующий шаг возможен только после отчёта.

### Принцип STOP
- STOP используется как обязательный сигнал остановки (см. режим минимальных инструкций).
- После STOP требуется сверка с KB перед продолжением.

### Принцип каноничности решений
- В KB фиксируются только:
  - архитектурные решения,
  - границы модулей,
  - изменения workflow.
- Тактические детали и обсуждения в KB не заносятся.
- При конфликте правил допускается только:
  - обновление,
  - либо замена старого правила новым.
  Параллельные противоречивые политики запрещены.

### Принцип диагностики
- Перед любыми изменениями требуется диагностика.
- Гипотезы без подтверждения не считаются основанием для решений.

## Архитектурное эхо
- Перед началом проверки кратко повторить архитектурную цель задачи: «что и почему меняем».
- Эхо фиксируется в первом сообщении тестировщику и в этом документе при необходимости.
- Без подтверждённого эха — никаких действий/кода.

## «Не та корова» (контроль контекста)
- Если тестировщик проверяет не тот модуль/страницу/роль — остановить и вернуть к правильной цели.
- Стоп-фраза: «Стоп. Это не та корова.»
- Признак: шаги теста не соответствуют заявленному сценарию.
- Действие: одним сообщением указать правильный сценарий и ссылку на запись в книге проекта.

## Каноническая фиксация
- Результаты теста фиксируются в одном месте: в задаче + ссылка на запись в книге проекта.
- Фиксации в чате не считаются; всё важное — только в KB.
- Любые новые наблюдения вносятся в:
  - [CANON/DECISIONS_LOG](DECISIONS_LOG.md)
  - [SPEC/RUNBOOKS/TEST_RELEASE](../SPEC/RUNBOOKS/TEST_RELEASE.md)

## Один шаг тестировщику
- Сообщения тестировщику должны содержать ровно один следующий шаг.
- Пример формата:
  - «Откройте страницу X, выполните действие Y, ожидаемый результат Z».
- Если нужен контекст — добавить ссылку на раздел в KB.

## Шаблон сообщения тестировщику
```
Цель: <краткая цель/сценарий>
Шаг: <ровно одно действие>
Ожидание: <точный результат>
Ссылка: <раздел книги проекта>
```
