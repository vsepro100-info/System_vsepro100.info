## Процесс разработки (канонический workflow)

В проекте используется строгий цикл взаимодействия ролей:

1) Архитектор формулирует задачу для Codex.
2) Codex реализует изменение и оформляет Pull Request (PR).
3) Codex предоставляет Summary выполненных изменений.
4) Тестировщик передаёт отчёт Архитектору.
5) Архитектор принимает решение:
   - разрешить merge,
   - либо отклонить PR,
   - либо запросить доработку.
6) После merge Архитектор указывает:
   - какие именно файлы / плагины необходимо загрузить на хостинг;
   - какие действия должен выполнить тестировщик.
7) Тестировщик выполняет действия и предоставляет отчёт.

Правила:
- Без PR изменения не считаются существующими.
- Без команды Архитектора запрещены любые изменения на хостинге.
- Архитектор — единственная точка принятия решений.
