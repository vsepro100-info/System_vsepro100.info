# Runbook деплоя

## Цель
Обеспечить предсказуемый деплой: от мержа до проверки на целевой среде.

## Предусловия
- Изменения зафиксированы в репозитории.
- Релевантные решения отражены в [CANON/DECISIONS_LOG](../../CANON/DECISIONS_LOG.md).

## Стандартный цикл после мержа
1. **Скачать ZIP репозитория**
   - Использовать актуальную ветку, содержащую мерж.
2. **Деплой на хостинг**
   - Развернуть содержимое ZIP на целевую среду.
   - Для плагинов путь деплоя: `/wp-content/plugins/`.
3. **Проверка**
   - Пройти [RUNBOOKS/TEST_RELEASE](TEST_RELEASE.md).
   - Проверить ключевые страницы из [SPEC/WEBINARS/PAGES](../WEBINARS/PAGES.md).
4. **Если эффекта нет**
   - Проверить кэш.
   - Проверить активность плагина.
   - Если плагина не видно в админке — это может быть НЕ баг, а не задеплоено/не туда скопировано/не обновлена папка.

## Типовые грабли и как их избежать
- **Забыли обновить зависимый модуль** → всегда сверять с [CANON/MODULE_BOUNDARIES](../../CANON/MODULE_BOUNDARIES.md).
- **Изменение есть в коде, но не появилось в интерфейсе** → очистить кэш.
- **Доступы не совпадают с ожиданиями** → проверить роли в [CANON/ROLES](../../CANON/ROLES.md).
- **Новые шорткоды не работают** → сверить регистрацию в [SPEC/WEBINARS/PAGES](../WEBINARS/PAGES.md).
