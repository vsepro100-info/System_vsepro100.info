# Правила работы с тестировщиком

## Архитектурное эхо
- Перед началом проверки кратко повторить архитектурную цель задачи: «что и почему меняем».
- Эхо фиксируется в первом сообщении тестировщику и в этом документе при необходимости.

## «Не та корова» (контроль контекста)
- Если тестировщик проверяет не тот модуль/страницу/роль — остановить и вернуть к правильной цели.
- Признак: шаги теста не соответствуют заявленному сценарию.
- Действие: одним сообщением указать правильный сценарий и ссылку на запись в книге проекта.

## Каноническая фиксация
- Результаты теста фиксируются в одном месте: в задаче + ссылка на запись в книге проекта.
- Любые новые наблюдения вносятся в разделы:
  - [Известные проблемы](KNOWN_ISSUES.md)
  - [Лог решений](DECISIONS_LOG.md)
  - [Чек-лист тестирования](TESTING_CHECKLIST.md)

## Один шаг тестировщику
- Сообщения тестировщику должны содержать ровно один следующий шаг.
- Пример формата:
  - «Откройте страницу X, выполните действие Y, ожидаемый результат Z». 
- Если нужен контекст — добавить ссылку на раздел в книге проекта.

## Шаблон сообщения тестировщику
```
Цель: <краткая цель/сценарий>
Шаг: <ровно одно действие>
Ожидание: <точный результат>
Ссылка: <раздел книги проекта>
```
